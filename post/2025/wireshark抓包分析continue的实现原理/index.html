在前三篇文章中，我们已经成功在M1芯片的MacBook上安装了Docker，搭建了本地大模型服务（Ollama），并在VSCode中配置了Continue插件连接本地Ollama服务。

现在，我们来深入研究Continue插件的各种模式，并且会通过Wireshark抓包来分析Continue与LLM的交互过程，从而学习Agent的运行机制。。

Continue插件的三种核心模式

Continue插件提供了三种核心交互模式：Chat模式、Plan模式和Agent模式。

Chat模式

Chat模式是最基础的对话模式，允许用户与大模型进行多轮对话。这种模式类似于传统的聊天机器人，用户输入问题，模型返回回答。

Note

Chat模式主要处理用户的问题和模型的回复，不涉及复杂的代码编辑操作。

测试过程中，我会在同一个会话中提交多个问题，然后用Wireshark监听回环网口。

Chat模式抓包分析

使用Wireshark来抓取Chat模式的网络请求的步骤如下：

  1. 启动Wireshark，选择本地回环接口（lo0）
  2. 设置过滤条件：http and ip.src == 127.0.0.1 and ip.dst == 127.0.0.1 and tcp.port == 11434
  3. 在VSCode中使用Continue插件的Chat模式发送消息
  4. 在Wireshark中观察请求和响应

不难发现，这次请求的message包含的内容如下如下：

  1. system 角色的提示词；
  2. 会话中前文的user提问的内容；
  3. 会话中前文assistant的回答的内容；
  4. 本次用户提问的内容；

由此，我们就很容易发现，其实连续对话的实现原理如下：

  1. Continue首先在system prompt中定义好自己的角色；
  2. Continue插件从持久化存储中读取本次会话的历史消息；
  3. Continue插件将本次用户提问的内容拼接到会话历史；

最终将所有内容一次性灌给/api/chat接口，并得到assistant的回答。

至于/api/chat的底层，其实也只是把message拼装成一个完整的prompt，然后由LLM输出结果。

这种模式下：

  • 会话内容只存放在Continue这个Agent中；
  • LLM和Ollama只会看到最终聚合在一起的一个输入prompt，LLM只需要计算结果，由Ollama返回即可。

Plan模式

Plan模式是更高级的功能，它会先生成一个执行计划，然后逐步执行。

这种模式适用于需要多步骤完成的任务，比如重构整个文件或实现一个复杂功能。

Agent模式时序图

三种模式的差异分析

实现思路对比

模式核心思想适用场景复杂度交互方式
Chat简单问答代码解释、问题解答低单次请求-响应
Plan任务分解复杂功能实现、重构中多次请求-响应
Agent自主决策自动化编程、复杂问题解决高循环决策-执行-验证

为什么需要不同的模式？

  1. Chat模式：适用于简单的问答场景，用户明确知道自己要问什么，模型直接给出答案。

  2. Plan模式：适用于复杂任务，需要先规划再执行，避免盲目操作。这种模式让用户可以看到执行计划，增加透明度和可控性。

  3. Agent模式：适用于需要自主分析和决策的场景，模拟了程序员解决问题的思路：分析-计划-执行-验证-迭代。

网络交互差异

从抓包结果可以看出，三种模式在与Ollama的交互上存在显著差异：

  1. 请求结构：Chat模式的请求相对简单，Plan和Agent模式的请求包含更多上下文信息
  2. 请求次数：Chat模式通常是1对1的请求响应，而Plan和Agent模式会发起多次请求
  3. 响应内容：Agent模式的响应通常包含更多元数据，用于指导下一步操作

实践案例：文件编辑测试

让我们通过一个具体的例子来测试这三种模式：

假设我们有一个简单的JavaScript函数需要重构：

1function calculateTotal(items) {
2  let total = 0;
3  for (let i = 0; i < items.length; i++) {
4    total += items[i].price;
5  }
6  return total;
7}

Chat模式测试

在Chat模式下，我们直接询问"如何优化这个函数？"，模型会给出建议，但不会直接修改代码。

Plan模式测试

在Plan模式下，我们请求"重构这个函数以提高性能"，模型会先生成一个计划，然后给出每一步应该做的事情，指导我们完成重构。

Agent模式测试

在Agent模式下，我们请求"自动优化这个函数"，Agent会分析代码、制定策略、执行修改并验证结果。

深入分析：Continue的实现机制

会话管理

Continue插件通过维护会话状态来实现多轮对话。每个会话包含：

  1. 对话历史记录
  2. 当前代码上下文
  3. 用户偏好设置

上下文构建

在每次请求前，Continue会构建包含以下信息的上下文：

  1. 当前编辑的文件内容
  2. 选中的代码片段
  3. 相关文件的引用
  4. 之前的对话历史

模型选择策略

Continue根据任务类型自动选择合适的模型：

  1. 简单补全使用轻量级模型
  2. 复杂推理使用大模型
  3. 代码理解使用专门的代码模型

结语

通过深入分析Continue插件的三种模式，我们了解了现代AI编程助手的实现机制：

  • Chat模式提供基础的问答能力；
  • Plan模式实现任务分解和有序执行；
  • Agent模式则实现了更高级的自主决策能力。

通过Wireshark抓包分析，我们能够清晰地看到这三种模式在与LLM交互时的差异，这有助于我们更好地理解AI助手的工作原理，并在实际开发中更有效地利用这些工具。
